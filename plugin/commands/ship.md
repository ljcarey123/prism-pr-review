# Ship — Generate Report + Create GitHub PR

Full shipping workflow: analyse the branch, generate the `.pr/index.html` report, commit it, push, and open a GitHub PR with the AI-generated summary as the PR body.

**Arguments:** "$ARGUMENTS"
- First positional arg = target branch (default: `main`)
- `--draft` = open as a draft PR

---

## Step 1 — Pre-flight checks

Run these in parallel:
```bash
git branch --show-current          # confirm we're on a feature branch
git status --short                 # check for uncommitted changes
gh auth status                     # confirm gh CLI is authenticated
```

- If on `main`/`master`/`trunk`, stop and tell the user to switch branches.
- If there are uncommitted changes, warn the user and ask if they want to continue anyway (the report will reflect the committed state).
- If `gh` is not authenticated, stop with instructions: `gh auth login`.

Parse arguments:
- Target branch: first word in `$ARGUMENTS` that doesn't start with `--`, else default `main`
- Draft flag: `true` if `--draft` appears in `$ARGUMENTS`

---

## Step 2 — Generate the PR review report

Follow the full review workflow (same as `/review-pr`):

1. Call `get_pr_data({ target_branch: "<target>" })` to collect git data
2. Explore codebase structure to understand component boundaries
3. Analyse the diff — classify major/minor changes, risk, components, test coverage
4. Build the PRAnalysis JSON
5. Call `generate_report({ analysis: "<json>", output_path: ".pr/index.html", open_browser: true })`

Save the analysis object in memory — you'll need `summary` and `overall_risk` for the PR description.

---

## Step 3 — Commit the report

```bash
git add .pr/index.html
git commit -m "chore: add PR review report"
```

If `.pr/index.html` was already committed and unchanged, skip this step.

---

## Step 4 — Push the branch

```bash
git push -u origin HEAD
```

If the push fails due to a protected branch or auth issue, stop and surface the error clearly.

---

## Step 5 — Create the GitHub PR

Build the PR body from the analysis:

```markdown
## Summary

{summary from analysis — 2-3 sentences}

## Changes

**Risk level:** {overall_risk}

### Major changes ({count})
{bullet list: file — description [risk badge]}

### Minor changes ({count})
{bullet list: file — description}

## Impacted components
{bullet list: component name — description}

## Test coverage
**Assessment:** {assessment}
{notes}
{list of untested files if any}

---
*Generated by [prism](https://github.com/your-org/prism) 🔮*
```

Then create the PR:
```bash
gh pr create \
  --title "<concise PR title from the summary>" \
  --body "<formatted body above>" \
  --base "<target_branch>" \
  {--draft if draft flag set}
```

For the `--title`: derive a concise (≤70 chars) imperative title from the summary (e.g. "Add JWT authentication to API endpoints").

After creation, output the PR URL so the user can open it.

---

## Step 6 — Done

Report back:
- PR URL
- Risk level
- Count of major / minor changes
- Whether the report was committed
- Direct link to `.pr/index.html` (relative path for the branch)
